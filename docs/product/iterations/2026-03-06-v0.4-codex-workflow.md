# Memory OS Iteration: v0.4 Codex Workflow

## Basic Info

- Iteration: `v0.4`
- Status: `completed`
- Owner: Lin
- Date: 2026-03-06

## Why This Iteration Exists

v0.3 proved that Memory OS can be registered as an MCP server in Codex CLI and that Codex can call Memory OS tools successfully.

The remaining friction is workflow friction:

- the user still needs to remember how to inject context
- the user still needs to manually frame the Codex session

## Context

- Memory OS now supports CLI export, local API, MCP, and Codex MCP registration.
- Codex can already call the `memory-os` MCP server.
- The next step is to make the common "start a Codex task with project context" path faster.

## Goal

Create a Codex workflow wrapper that automatically assembles Memory OS context and launches Codex with that context as the task brief.

## Scope

In scope:

- add a Codex wrapper script
- support project-based context injection
- support a print-only mode for inspection
- support a non-interactive Codex execution path
- document usage

Out of scope:

- deep Codex config automation
- project presets
- browser integrations
- automatic write-back from Codex into Memory OS

## User Value

This iteration reduces the steps needed to start a real Codex session with aligned user and project context.

Instead of:

- exporting or fetching profile
- exporting or fetching project context
- manually combining them
- manually phrasing the task

the user can start from one wrapper command.

## Product Decisions

- The wrapper should reuse existing local API and export logic.
- The wrapper should remain thin and composable.
- Print mode is required so users can inspect exactly what Codex receives.
- Non-interactive execution is enough to validate usefulness before considering interactive wrappers.

## Functional Changes

- add Codex workflow wrapper
- support `--project`
- support `--print`
- support `--exec "<task>"`
- document example workflows

## Technical Changes

- reuse existing agent context composition logic
- shell out to `codex exec` for non-interactive runs
- avoid hard-coding vendor-specific config mutations
- add tests for prompt assembly and wrapper behavior

## Validation Plan

- run the wrapper in print mode and inspect generated briefing
- run the wrapper in `codex exec` mode with a real project
- verify that Codex responds correctly using injected Memory OS context

## Risks

- wrapper may duplicate too much of existing script logic
- prompt assembly may become too verbose
- Codex task behavior may vary depending on local config

## Open Questions

- should future versions support interactive `codex` mode in addition to `codex exec`?
- should project presets be introduced once common workflows stabilize?

## Exit Criteria

- wrapper works in print mode
- wrapper works in `codex exec` mode
- at least one real Codex workflow is completed successfully
- documentation explains the new path clearly

## Result Summary

- Added `scripts/codex-memory.mjs` as a thin Codex workflow wrapper.
- The wrapper reuses the local API to fetch:
  - profile export
  - project context export
- The wrapper supports:
  - `--project`
  - `--print`
  - `--exec "<task>"`
  - passthrough Codex args after `--`
- Added automated tests for:
  - print mode
  - forwarding the assembled prompt to `codex exec`
- Ran a real end-to-end validation:
  - started `mem serve`
  - ran `node scripts/codex-memory.mjs --project demo_001 --exec "..."`
  - Codex returned `Memory OS`, confirming the wrapper can launch a real task with injected Memory OS context

## Validated Effect

- Starting a Codex task now takes one wrapper command instead of manually exporting and combining context.
- The wrapper keeps prompt assembly consistent with the same Memory OS export logic used elsewhere.
- Codex integration now supports two practical paths:
  - on-demand MCP tool usage
  - upfront briefing injection via `codex-memory`

## Next Iteration Recommendation

If this iteration proves useful, the likely next step is:

- adding project presets and saved workflow shortcuts
- or extending the same wrapper approach to other host tools such as Claude Code
