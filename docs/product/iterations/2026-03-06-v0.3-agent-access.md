# Memory OS Iteration: v0.3 Agent Access

## Basic Info

- Iteration: `v0.3`
- Status: `completed`
- Owner: Lin
- Date: 2026-03-06

## Why This Iteration Exists

v0.2 proved that Memory OS can expose personal context through a local API and that an external script can consume project context.

The next product step is to make that access layer more usable for real agents and more interoperable with MCP-capable clients.

## Context

- v0.x remains scoped to the personal context layer.
- v0.2 introduced a local read-only HTTP API.
- A minimal external script has already validated direct project-context fetching.
- The next bottleneck is not storage or schema, but agent-facing access patterns.

## Goal

Make Memory OS easier to plug into real agent workflows by:

- adding practical adapter scripts for agent-style usage
- exposing an MCP-compatible server that maps Memory OS context into MCP primitives

## Scope

In scope:

- add one or more agent adapter scripts that consume the local API
- support prompt-ready context assembly for agent workflows
- add an MCP-compatible server over stdio
- expose Memory OS context as MCP resources, prompts, and/or tools
- validate both adapter and MCP flows end to end

Out of scope:

- browser extension
- cloud sync
- automatic memory writing by agents
- full growth system
- full conversation archival

## User Value

This iteration improves Memory OS in two ways:

- direct practical use in terminal and coding-agent workflows
- interoperability with MCP-based clients without custom one-off integrations

## Product Decisions

- Keep the product boundary inside the personal context layer.
- Prefer read-first integration over autonomous write access.
- Use a narrow MCP surface that maps closely to existing Profile, Project, and Insight entities.
- Adapter scripts should stay lightweight and composable rather than becoming a full orchestration system.

## Functional Changes

- add agent adapter script(s)
- add MCP server entrypoint
- expose core memory context through MCP primitives
- document usage and validation steps

## Technical Changes

- reuse existing local API and exporter logic
- implement stdio-based MCP transport first
- avoid coupling MCP implementation to GUI or Electron
- add tests for MCP lifecycle and core resource/prompt/tool access

## Validation Plan

- run an adapter script against the local API and inspect output
- run the MCP server and verify initialize plus at least one resource/prompt/tool flow
- confirm that the new access methods reuse existing context rather than duplicating business logic

## Risks

- overbuilding the MCP layer too early
- creating adapter scripts that are too tool-specific to stay maintainable
- introducing an access layer that diverges from existing export semantics

## Open Questions

- Should Memory OS expose prompts, resources, tools, or all three in MCP?
- Which agent workflow should become the reference integration first?
- How much prompt shaping belongs in adapter scripts versus export logic?

## Exit Criteria

- at least one practical agent adapter script works end to end
- MCP server initializes successfully and returns useful Memory OS context
- tests cover the main access paths
- iteration results are documented clearly

## Result Summary

- Added practical agent adapter scripts:
  - `scripts/compose-agent-context.mjs`
  - `scripts/prepare-claude-code-context.mjs`
  - `scripts/prepare-codex-context.mjs`
  - `scripts/prepare-terminal-agent-context.mjs`
- Added Codex-specific MCP integration scripts:
  - `scripts/run-memory-os-mcp.mjs`
  - `scripts/install-codex-mcp.mjs`
- Added `mem mcp` as an MCP-compatible stdio server.
- Exposed Memory OS through MCP resources, prompts, and tools, reusing existing storage and exporter logic.
- Added automated tests for:
  - agent context composition
  - MCP initialize flow
  - MCP prompt/tool access
  - Codex MCP registration in an isolated HOME
- Manually validated two end-to-end flows with isolated local data:
  - local API -> Claude Code/Codex adapter scripts
  - MCP initialize -> `get_project_context` tool call
- Registered `memory-os` into the real local Codex CLI configuration and verified it via `codex mcp list` and `codex mcp get memory-os --json`.
- Ran a real `codex exec` session that called the `memory-os` MCP tool `get_project_context` for project `demo_001`, confirming host-level integration works beyond protocol-level tests.

## Validated Effect

- Memory OS can now be consumed in three access patterns:
- Memory OS can now be consumed in four access patterns:
  - direct CLI export
  - local HTTP API
  - MCP-compatible stdio server
  - Codex CLI via registered MCP server
- Agent workflows no longer need to manually reconstruct user profile and project state from raw memory data.
- The same exporter logic is reused across CLI, HTTP API, adapter scripts, and MCP, reducing product drift.

## Next Iteration Recommendation

If this iteration proves useful, the likely next step is:

- narrower agent-specific integrations based on the user's real preferred tools
- or MCP refinement based on actual client usage patterns and prompt/resource/tool ergonomics
