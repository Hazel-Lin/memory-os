# Memory OS Iteration: v0.2 Local API

## Basic Info

- Iteration: `v0.2`
- Status: `completed`
- Owner: Lin
- Date: 2026-03-06

## Why This Iteration Exists

The current product can store and export personal context through CLI, but it is still cumbersome for other tools and agents to consume the data directly.

If Memory OS is meant to be a cross-tool personal AI context layer, it needs a stable local interface beyond terminal export commands.

## Context

- v0.1 has validated the basic CLI flow for profile, project, insight, and export.
- The product direction has been narrowed to the personal context layer.
- GUI exists but should remain secondary.
- The next product bottleneck is access, not more entities.

## Goal

Add a local read-first API so external tools and agents can retrieve personal context without requiring manual copy-paste from CLI output.

## Scope

In scope:

- define the first local API surface
- expose read-only endpoints for profile, projects, and insights
- expose a project context endpoint aligned with current export logic
- document usage examples for external tools

Out of scope:

- write endpoints
- authentication system
- browser extension
- growth system or reflection features
- cloud sync

## User Value

This iteration reduces friction when connecting Memory OS to:

- terminal agents
- coding agents
- future MCP integrations
- automation scripts

Instead of manually running export commands every time, users and tools can request stable local context programmatically.

## Product Decisions

- The API should be local-only in this iteration.
- Read-only access is enough to validate integration demand.
- The initial API should stay narrow and map closely to current entities.
- Export-oriented context remains important, so API responses should support both raw data and prompt-ready views where useful.

## Functional Changes

- add a local HTTP server entrypoint
- add `GET /profile`
- add `GET /projects`
- add `GET /projects/:id`
- add `GET /projects/:id/context`
- add `GET /insights`
- add `GET /insights?project=<id>`

## Technical Changes

- add local API module
- reuse existing storage and exporter logic
- keep storage format unchanged
- avoid coupling API behavior to GUI
- add tests for endpoint behavior

## Validation Plan

- run the local API and query all endpoints from terminal
- connect one external script or agent to consume project context
- verify that context access is faster than CLI copy-paste in at least 3 real workflows

Success looks like:

- endpoint responses are stable
- project context can be consumed without manual editing
- the user prefers API access in at least some repeat workflows

## Risks

- API may duplicate CLI behavior without enough real workflow gain
- response format may be too raw or too prompt-specific
- scope may drift toward full integration infrastructure too early

## Open Questions

- Should `/projects/:id/context` return raw structured JSON, prompt-ready text, or both?
- Should the API start as plain HTTP only, or immediately consider MCP alignment?
- Does the user need a lightweight daemon workflow, or is an on-demand server enough?

## Exit Criteria

- local API can be started reliably
- all read endpoints return valid data
- at least one real external workflow uses the API
- product boundary remains within the personal context layer

## Result Summary

- Added a local read-only HTTP API behind `mem serve`.
- Shipped endpoints for `GET /health`, `GET /profile`, `GET /projects`, `GET /projects/:id`, `GET /projects/:id/context`, and `GET /insights`.
- Reused existing storage and exporter logic so the API returns both raw structured data and prompt-ready exports where useful.
- Added an integration test that starts the real CLI server and validates endpoint behavior.
- Added `scripts/fetch-project-context.mjs` as a minimal external consumer for project context.
- Validated a real external workflow by starting `mem serve` with isolated local data and fetching project context through the script.
- Remaining work is broader validation with one or more real agents in day-to-day usage.

## Next Iteration Recommendation

If v0.2 confirms real cross-tool usage, the likely next step is either:

- `v0.3 import/export and integration refinement`

or

- `v0.3 MCP-compatible access layer`
